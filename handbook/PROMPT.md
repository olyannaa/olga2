# PROMPT: Справочник пользователя по ISET Group

Ты — технический писатель и продуктовый аналитик. На основе документации и логики репозитория спроектируй и заполни справочник пользователя для всего сервиса. Справочник доступен через меню «Справочник» и должен объяснять, как пользоваться системой для каждой роли. Нужно разобрать абсолютно все UI действия и правила, которые реально работают в текущем коде.

## Источники истины
1) Код UI: `work-stream-alchemy-main/src/**`
2) Код backend: `backend/src/**` и миграции `backend/migrations/**`
3) Документация: `memory-bank/**`, `planning/v3/**`, `planning/v4/**`, `README.md`

Если требования расходятся, приоритет — фактическое поведение кода. Если функциональность не реализована, явно так и укажи.

## Роли
Учитывай различия для ролей: `admin`, `gip`, `executor`, `accountant`. Также учти проектные роли: `manager`, `lead_specialist`, `executor`, `accountant`, а также особенности scoped admin и флаг `canApproveSubcontracts`.

## Обязательные разделы и правила
Для каждого раздела:
- Какие действия доступны в UI (создать/изменить/удалить/переместить/фильтровать/экспортировать и т.д.).
- Какие правила и ограничения действуют (RBAC, статусы, блокировки).
- Чем отличаются роли (включая проектные роли).
- Что происходит в данных/процессах (например, авто‑создание задач, ограничения по субподрядам).
- Отдельно указать не реализованное/заглушки.

Обязательно укажи примеры:
- ГИП может создавать проекты и при этом может быть назначен в проекте как исполнитель.
- Исполнитель видит только свои задачи и может менять их статус.
- Админ с `canApproveSubcontracts` видит «Заявки» и может согласовывать.

## Дополнительные требования
- Не придумывай функциональность, которой нет в коде.
- Если действие запрещено, покажи причину и роль, для которой это ограничение действует.
- Укажи все статусы задач/проектов, ограничения на смену статусов и правила субподряда.
- Укажи, какие страницы и действия работают только для определенных ролей.
